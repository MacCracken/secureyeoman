# ADR 167 — Native Clients: Tauri v2 Desktop + Capacitor v6 Mobile

**Date**: 2026-02-28
**Status**: Accepted
**Deciders**: Core team
**Related**: ADR 159 (Dashboard Performance), Phase 91

---

## Context

SecureYeoman's REST + WebSocket API is stable (Phases 79–90) and the compiled dashboard SPA
(`packages/dashboard/dist`) already runs well in a browser. The roadmap lists desktop and mobile
native clients as "Future / Demand-Gated". This ADR records the technology choices for
scaffolding both shells without duplicating any UI or API code.

---

## Decision 1 — Desktop: Tauri v2 (not Electron)

### Rationale

| Criterion | Tauri v2 | Electron |
|-----------|----------|----------|
| Binary size | ~5–15 MB | ~150–250 MB (ships Chromium) |
| Renderer | OS WebView (WKWebView / WebView2 / WebKitGTK) | Bundled Chromium |
| Runtime | Rust process | Node.js + Chromium |
| Memory at idle | ~30–60 MB | ~150–300 MB |
| Shell access | `tauri-plugin-shell` (sandboxed allow-list) | Full Node.js `child_process` |
| Update mechanism | Tauri updater plugin | electron-updater |
| Security model | Rust process isolation, explicit IPC allow-lists | Full Node.js exposed to renderer |

SecureYeoman's security-first philosophy demands minimal attack surface. Tauri's explicit IPC
allow-list model means no unintentional shell access from web content. The Rust process model
provides OS-level isolation between the shell and the renderer.

### Implementation

The Tauri shell in `packages/desktop/src-tauri/` loads `packages/dashboard/dist` as its
`frontendDist`. In dev mode it proxies `http://localhost:3000` (Vite dev server). No dashboard
code changes are required.

Enabled plugin: `tauri-plugin-shell` — allows the shell to open external URLs in the default
browser (OAuth redirect flows, documentation links). No other IPC surface is exposed.

### Consequences

- macOS, Windows, and Linux ship from the same codebase.
- App Store submission (Mac App Store, Microsoft Store) requires additional notarisation and
  packaging steps not covered by this scaffold.
- Icons must be supplied before `tauri build` runs (see `src-tauri/icons/README.md`).
- Rust toolchain required for contributors building the desktop app.

## Decision 2 — Mobile: Capacitor v6 (not React Native)

### Rationale

| Criterion | Capacitor v6 | React Native |
|-----------|-------------|--------------|
| UI layer | Existing Vite SPA (zero duplication) | New React Native components |
| Dev ramp-up | None — web devs ship immediately | Requires RN ecosystem knowledge |
| Feature parity | 100% — all 100+ API endpoints work | Must re-implement feature by feature |
| Live-reload | Set `server.url` to local Vite dev server | Metro bundler |
| Native feel | Capacitor plugins (haptics, push notifications, secure storage) | Built-in |
| Bundle size | ~5 MB shell + web assets | Larger JS bundle, no web asset reuse |

The zero-duplication argument is decisive: the dashboard SPA already implements the full feature
set. Capacitor wraps it as a native WebView app and adds native plugin access incrementally
(push notifications, biometric auth, secure storage, haptics).

### Implementation

`packages/mobile/capacitor.config.ts` points `webDir` at `../dashboard/dist`. After building
the dashboard, `npx cap sync` copies assets into `ios/` and `android/`. The `ios/` and
`android/` directories are generated by `npx cap add ios` / `npx cap add android` and are not
committed (`.gitignore`).

### Live-Reload Dev Workflow

Point `server.url` in `capacitor.config.ts` at the local Vite dev server (`http://192.168.x.x:3000`).
The mobile WebView fetches assets from the dev server — hot module replacement works natively.

### Consequences

- Xcode required for iOS builds; Android Studio for Android builds.
- `ios/` and `android/` directories must be regenerated on a fresh checkout before building.
- App Store submission requires Apple Developer Program membership; Google Play requires a
  Google Play Developer account.
- `@capacitor/push-notifications`, `@capacitor/biometric-auth`, `@capacitor/secure-storage`
  are not installed in this scaffold; add as needed.

---

## Rejected Alternatives

### Flutter

Full widget re-implementation required — contradicts the zero-duplication goal. Dart is a
new language for the team. Rejected.

### PWA (Progressive Web App)

Already possible — any modern browser can install the dashboard as a PWA. Does not provide App
Store discoverability or deep native plugin access. PWA support is complementary, not a
replacement for native shells.

### Ionic + Capacitor (Ionic Framework)

Ionic wraps Capacitor but adds its own UI component library. Since SecureYeoman already has a
full UI (Tailwind + Radix), the Ionic layer would conflict. Plain Capacitor is lighter.

---

## Status

Scaffold complete. Native platform directories (`ios/`, `android/`) are generated by Capacitor
CLI and not tracked in git. Real icon assets, code-signing certificates, and App Store
configurations are out of scope for this phase.
